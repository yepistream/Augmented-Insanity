# Known Issues


### **NOTE**: *This is mostly for me to gack at, as a kind of map for the issues im trying to fix, as such I'm sad to annouce that I have used LLM to help me generate this page from a plain .txt to a nicer markdown file.*

Bugs I know about and have not yet fixed. Each entry lists the symptom,
the root cause (if known), the impact, and a tracking note.



## 11x lifecycle hook duplication

**Symptom.** `aug_onModuleLoad`, `aug_runtimeInit`, `aug_onConnect`, and the
generated IPC dispatch hooks (e.g. `xrCreateSession`,
`aug_spaceCreateSemanticIds`) each fire 11 times per logical event, visible
in `adb logcat -s "Aug-Ins:*" "AugIns.*:*"`.

**Where.** `src/xrt/augins/augins_lifecycle.cpp::fan_out` and the IPC
dispatch generated by `src/xrt/ipc/shared/proto.py`. Investigation in
progress.

**Root cause.** Not yet identified. `augins::launch()` is CAS-guarded and
prints "Aug-Ins launching" exactly once per process; the duplication is
elsewhere -- likely either `g_loaded_modules` accumulating duplicate
entries despite "Loaded module" logging once per module, or `fan_out` being
invoked 11 times by some path we have not yet located.

**Impact.** 11x IPC overhead per call, visible noise in logs. Functionally
the runtime still works because the hooks are idempotent (they read or
write the same reply 11 times). Modules whose hook bodies have side
effects need to be aware of the multiplication.

**Tracking.** `src/xrt/augins/augins_lifecycle.cpp` and the post-Mercury
stabilization plan; bisection steps documented there.

## Mercury init briefly fights ARCore VIO

**Symptom.** When the Mercury hand-tracking module starts (lazy-subscribed
on the first `xrLocateHandJointsEXT`), ARCore VIO emits 1-2
"FeatureExtraction taking too long" warnings. Tracking quality recovers
once Mercury inference settles.

**Root cause.** First-frame ONNX inference and TensorFlow Lite XNNPack
delegate spin-up are CPU-heavy. With Mercury's worker thread niced to +10
and frame intake decimated to 10 Hz (after init), steady-state CPU
contention is not an issue, but the first second is unavoidable.

**Impact.** Brief drop in head-pose tracking precision during the first
few seconds after Mercury starts. Does not affect modules that do not
trigger Mercury (head-pose-only clients).

**Tracking.** [Mercury Module Reference](Mercury-Module-Reference.md).
Mitigations already in place: `setpriority` nice +10 on the worker
thread; `kDecimateEveryN = 3` frame decimation in
`samples/augins-mercury-handtracking-arcore/mercury_arcore_module.cpp`.

## Misleading `In roles: <none>` log

**Symptom.** During service startup the log emits

```
monado.p_create_system: In roles:
    hand_tracking.unobstructed.left:  <none>
    hand_tracking.unobstructed.right: <none>
    ...
```

even when a hand-tracking-providing module is installed and working.

**Root cause.** `augins_stub_xdevs_install` installs the stub xdevs AFTER
`xrt_instance_create_system` has already logged the populated roles. The
log is correct at the time it fires; the runtime state is updated
immediately afterward.

**Impact.** Cosmetic. Hand tracking works regardless of this log.

**Tracking.** [Stub Xdev Factory](Stub-Xdev-Factory.md). Resolves when
the stub-xdev refactor moves installation into the system-builder
pipeline.

## Modules cannot ship Java code

**Symptom.** A `.augins` zip can contain native code (`.so`) and assets,
but no Java/Kotlin classes that the runtime APK can load.

**Root cause.** No DEX-loading mechanism in the runtime APK yet. Modules
currently can only call into the JVM via the host-API-provided
`get_vm()` / `get_context()`, which limits them to whatever Java classes
are already in the runtime APK or the system.

**Impact.** A module that needs a Java helper class (e.g. a USB-camera
bridge) must currently arrange for that class to be present in the
runtime APK. The Mercury USB module is blocked on this.

**Tracking.** [Roadmap](Roadmap.md) -- "Module-provided Activities and
DEX-loading".

## ARCore JNI-attach warnings

**Symptom.** Logcat emits

```
jni_helper.cc:110] Attempt to get JNIEnv* on thread not attached to JVM
```

a few times during ARCore session startup.

**Root cause.** ARCore-internal worker threads (depth estimation, ML
feature tracking) attempt to call back into the JVM without first
attaching. Not on any thread Aug-Ins owns.

**Impact.** None observed. ARCore's `jni_helper` warns and falls back to
the no-Java code path, which is what we want -- the calling subsystems
(depth estimation) are not used by the head-pose module.

**Tracking.** Closed as "noise, not a bug to fix on our side". If a
future module enables ARCore depth features, this will become relevant.
